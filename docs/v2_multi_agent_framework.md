# 智能分镜系统研发记录 V2.0：多智能体协作流 (Multi-Agent Framework)

## 1. 架构定位
在 V1.0 (纯提示词工程) 出现严重的幻觉、“金鱼记忆”光照错位等问题后，我们抛弃了单一 LLM 大包大揽的设计。V2.0 将“分镜绘制”这个极具艺术与工程难度的任务，**按照人类片场的流水线职位结构，解耦为了四个独立运行的 Agent。**

## 2. 系统核心 Agent 组件定义

### 🎬 Agent 1: 导演/编剧 Agent (Script & Director Agent)
* **核心职责**：剧作拆借与导演机位安排。
* **工作流**：接收一句话剧本 -> 扩写出 10-20 镜的结构化叙事 -> 针对每一镜，基于影视语言理论（建立景、越轴规则等）为其分配镜头参数（如 `Extreme Wide Shot` 等）-> 将文本字典传给下一个 Agent。

### ✍️ Agent 2: 提示词炼丹师 (Prompt Synthesizer Agent)
* **核心职责**：解决特征丢失和光影错误。把“人话”翻译成生图 AI 听得懂的“咒语”。
* **工作解决机制**：
  1. **实体特征追踪**：维护一个角色外貌的基础档案（如 `A boy with red hair, wearing a white shirt`），遇到当前场次的特定描述时仅在这个图谱上打补丁（如 `shirt stained with mud`），并在每一镜生图前，将补丁与基础档案拼装发送，杜绝特征丢失。
  2. **环境光解耦隔离**：将描写“场景/光影”的词与“主体角色”的主词强制分离注入。

### 🎨 Agent 3: 渲染执行 Agent (Render & Execution Agent)
* **核心职责**：苦力发包机器。不负责思考，只接管 Agent 2 发来的提纯后 Tag 表盘和渲染策略，负责请求稳定的第三方绘图引擎。
* **工程挑战解法**：
  * **场景空间坍塌应对**：负责先生成或调用空无一人的空旷场景基准底图（Base Environment Diagram），再将后续的角色控制网融进去。
  * **极限光影介入**：若遇到日夜变换/进入山洞等地带，负责唤醒 IC-Light 或类似模型进行光影重构。

### 👁️ Agent 4: 视觉一致性稽查 Agent (Consistency Vision Checker Agent)
* **核心职责**：AI 自纠错验收台。
* **机制**：通过接入具备多模态的视觉模型（如 Gemini Vision），接管 Agent 3 返回的渲染图，将其与 Agent 1 发布的导演要求进行像素与逻辑层面的 `Cross-Check`。
  * *判断项*：人数是否塞多？衣服特征是否扭曲错位？
  * 发现异常立刻打回重画。

## 3. 面临的瓶颈与局限 (Limitations of Multi-Agent)
表面上看，流水线极大地提高了系统的专业下潜度，也是当时学术界的标配。但在真实落地开发时遇到了几个痛点：

1. **沟通损耗与数据墙**：由于各 Agent 间以字符串/JSON在内存链中穿梭“开门关门”，系统架构极为臃肿死板，出错率增加。
2. **缺乏拔插弹性 (No Pluggability)**：每个 Agent 的技能包（Skills）都是与当前生图平台强绑定的死代码。今天这套管线对接了 Midjourney API，如果明天想要换成跑本地显卡的 ComfyUI，四大 Agent 都要改底层代码。
3. **把“工具”当“智能体”**：系统将一些纯机械工程（如连接 API、检验图片像素等纯函数）包装成 Agent 显得多此一举，缺乏真正的中央调度智慧。

## 4. 进化方向
V2 架构虽然理念绝佳地补齐了所有视觉盲区，但工程代码的死板促使我们抛弃“接力棒”式的 Multi-Agent 范式，转而走向以“顶层认知技能”与“底层开放协议工具箱”上下分离的 V3.0 (MCP Server) 架构。
