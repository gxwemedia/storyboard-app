# 智能分镜系统研发记录 V1.0：基线提示词工程架构 (Baseline Prompt Engineering)

## 1. 架构定位
这是通过大语言模型直接驱动生图 API 的最早期架构，也是行业内最常见的“初级套壳”方案。

## 2. 系统工作流 (Workflow)
系统由单一的 **大语言模型 (如 GPT-4 / Gemini 1.5 Pro)** 和单一的 **图像生成 API (如 Midjourney API / SD WebUI)** 构成。

1. **用户输入**：用户提供基础的剧本大纲或一段故事。
2. **大词典提示词展开**：LLM 接收到用户的剧本后，通过内部的 System Prompt，将剧本“翻译”或“扩写”成一连串带有图像描述标签（Tags）的 Prompt 组。
   * *示例输出*：`shot 1: a dark cave, a brave knight holding a torch, high quality, masterpiece...`
3. **批量 API 调用**：代码将这批 Prompt 放入一个 `for` 循环中，依次调用生图 API，下载返回的图片。
4. **合并结果**：最终将这些分散的图片拼接成一个分镜表返回给用户。

## 3. 面临的致命工程盲区 (Critical Issues)
在这个纯纯依赖“大力出奇迹”的阶段，系统暴露出极端严重的问题，导致其无法用于实际生产：

### 3.1 角色特征的“布朗运动” (Character Inconsistency)
如果没有对角色进行强约束（如 LoRA 或深度图绑定），仅靠自然语言描述（如 “金发碧眼的骑士”），生图模型每一帧画出的骑士长相、衣服款式和盔甲纹理都会完全不同。这在追求连贯叙事的漫画或影视分镜中是致命伤。

### 3.2 空间场域的坍塌 (Spatial Collapse)
在描述室内场景时，上一帧相机位于角色前方，背景是一扇窗；下一帧反打镜头，背景应该变成门。但大模型和生图 API 之间没有建立“三维空间意识”，导致场景布局随机改变，无法建立统一的世界观。

### 3.3 LLM 上下文撑爆与遗忘 (Token Explosion & Amnesia)
当分镜数量超过 20 镜时，LLM 极易忘记第 1 镜里主角“受了伤”的设定。生图的 Prompt 又会恢复成完好无损的初始状态。

## 4. 结论与下一步方向
V1.0 架构只证明了“AI 画分镜”的初步可行性。为了解决上述“不连贯”与“智商降级”的问题，系统必须走向**解耦**：我们需要专门负责记忆角色的模块、专门负责检查一致性的模块，由此引出了 V2.0 多智能体 (Multi-Agent) 架构的设计。
