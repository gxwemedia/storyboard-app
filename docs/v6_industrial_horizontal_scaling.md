# 智能分镜系统研发记录 V6.0：工业级横向支撑能力与元数据规范 (Horizontal Scaling & Metadata)

## 1. 痛点：一条流水线为何还称不上“工业化”？
在 V5 架构中，系统打通了“扩写 -> 设定 -> 分镜 -> 预演 -> 终图”的纵向 5 步流水线，成功将 Orchestrator 从盲盒抽卡机变为了“执行制片流水线进程”。
然而，真实的影视工业体系，除了流水线，更仰仗于**结构化数据流转、不可变的版本追溯、多工种协作分发**。如果没有横向系统的支撑，流水线里的数据极容易产生飘移或彻底乱套。因此，V6 聚焦于构建 4 个维度的横向骨架层，真正闭环工业标准。

## 2. 核心缺口补全规划 (The 4 Pillars of V6)

### 支柱 1：新增 Stage 0 (项目视觉圣经 Project Bible)
* **痛点**：在进行剧本扩写前，必须定死项目的基调。否则 AI 很容易写着写着从写实史诗变成了赛博朋克。
* **架构解法**：建立 `Ground Truth Level 0`。
  * **输入源**：导演的 Moodboard、风格片段、一句话故事、预算上限、禁忌元素词典。
  * **输出物**：固化的【世界观辞典】、【色彩基调卡 (Color Palette)】、【镜头规则集】。所有后续节点强制读取 Stage 0 圣经约束。

### 支柱 2：全链路中间态改造 (结构化分镜包 ShotSpec JSON)
* **痛点**：如果 Stage 3 仅仅输出“给人类阅读”的富文本或表格，后续的生图大模型依旧要重新理解自然语言（Loss 很大）。
* **架构解法**：强迫 Orchestrator 输出高度结构化的参数。所有的镜头信息必须落为机器完全可控的 `ShotSpec JSON` 实体。
  * 必须涵盖：镜号、景别 (Scale: Wide/MCU/CU)、焦段 (Lens: 24mm/50mm/135mm)、主光位 (KeyLight: Backlit/Rembrandt)、轴线锚点、连戏逻辑锁定。

### 支柱 3：变更传导与版本追踪 (Change Propagation & DAG)
* **痛点**：导演在 Stage 4 突然发现光影全错，或者在 Stage 5 突然后悔了想修改 Stage 1 的人物属性。现有的直线回退会导致下游工作全毁。
* **架构解法**：
  * **资产血缘网 (DAG 图)**：追踪每张图是由哪些 FaceID、哪行剧本生成的。
  * **爆炸半径控制**：如果导演仅修改了 Shot 02 的环境光参数，系统只废弃 Shot 02 后续的缓存，保全其他无依赖分镜资产。
  * **支持局部重绘 (Inpainting/ControlNet Mask)**，而不是动辄整张图重跑。

### 支柱 4：终版签发、拆解与包体归档 (Sign-off & Export Pipeline)
* **痛点**：网页前端的瀑布流只能看，制片团队无法拿到实用的生产物料。
* **架构解法**：在完成最终打磨和批注后，实现资产打包。
  * 导出可供外包动画公司的**分镜 PDF 剧本集**。
  * 导出供现场摄制组的 **镜头采购单 / 场表 CSV**。
  * 导出包含了所有 Lora 训练参考图、Prompt 历史、Midjourney Seed 的**核心元资产包 (Asset Archive)**。

## 3. 导演控制台界面重构 (Director Console UI)
纵向管线解决的是“流程对不对”，横向支撑层解决的是“数据稳不稳”，但真正让导演愿意每天盯着系统工作的，还是界面是否符合专业决策节奏。

旧式原型常见的问题是：把进度条、输入区、审核区、日志区、结果区简单上下堆叠。这样虽然“功能都在”，但实际会让导演在一个屏幕上同时处理**流程导航、内容判断、系统监控、结果验收**四种不同认知任务，造成视觉竞争与操作疲劳。

因此，V6 在数据模型之外，补充了一套**导演控制台信息架构**，原则不是“堆更多面板”，而是把不同认知任务放到不同的稳定区域中。

### 3.1 布局原则：把“当前决策”放在正中央
* **左侧 Workflow Rail（流程导航轨）**：只承载项目级输入和阶段导航，例如 Stage 0~5 进度、剧本大纲输入、启动/重置/导出入口。它负责告诉导演“我现在在哪、接下来去哪”，而不是让导演在这里做复杂判断。
* **中央 Director Workbench（导演工位）**：这是全界面的主舞台。无论当前是视觉圣经审批、文本扩写修订、ShotSpec 修改还是终版签发，所有核心判断都只在这里发生。这样做的好处是：导演眼睛永远知道该盯哪里。
* **右侧 Inspector Rail（监控/日志轨）**：把 MCP Server 状态、系统日志、执行反馈放到右侧，作为辅助信息存在。它们非常重要，但不应压过“导演审批”这个主任务。
* **底部 Output Dock（结果仓）**：最终渲染图不再与审批工作区抢位置，而是单独停靠在底部，形成“过程在上、结果在下”的明确视觉语义。

### 3.2 为什么这比传统原型更像工业系统
* **持久上下文不消失**：项目大纲、阶段导航、系统状态都稳定停留在固定区域，避免导演在流程推进后失去上下文。
* **审批动作单线程化**：中央工位一次只服务一个阶段，减少同时出现多个操作面板造成的决策噪音。
* **监控信息降噪**：日志和服务状态被压缩到右侧轨道，只在需要时阅读，不再干扰主线工作。
* **结果独立归仓**：终图区从“工作区的一部分”变成“交付仓”，更贴近制片流程中“出片后再审签”的真实节奏。

### 3.3 在原型中的具体落地
最新原型界面已经据此重构为一个更接近真实桌面工作站的结构：
1. 顶部 `Mission Header` 负责解释当前系统模式与当前产物类型。
2. 左侧 `Workflow Rail` 纵向放置步骤导航、项目输入与工作原则说明。
3. 中央 `Director Workbench` 永远留给当前阶段的 HITL 审核内容。
4. 右侧 `Inspector Rail` 集中显示系统监控、MCP 状态与 Orchestrator 日志。
5. 底部 `Output Dock` 独立承载最终分镜渲染流，形成明确的结果区。

这意味着，V6 不只定义了资产的结构化与可追溯，也定义了**人如何高效地驾驭这套系统**。

## 4. 架构结论
V5 定义了“如何一步步把戏拍完”的**行为模式 (State Machine)**。
V6 定义了“拍完的戏由谁保管、如何可溯源、怎么交接给其他部门”的**数据模型 (Data Model)**，以及“导演如何在单屏工作台上高效做出审签决策”的**交互模型 (Interaction Model)**。
至此，AIGC 系统方能真正登堂入室，从一个辅助绘图软件，跃升为**数字影视生产与资产管理总平台 (Production & Asset Pipeline)**。
